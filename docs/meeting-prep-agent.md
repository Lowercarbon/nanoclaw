# LC Meeting Prep Agent

A NanoClaw agent that delivers contextual meeting briefings to Lowercarbon Capital team members via Slack DM before calendar events.

## Architecture

The agent runs inside NanoClaw's Docker container with access to multiple data sources via 5 independent MCP servers:

- **Google** (`mcp-servers/google/`) — Calendar reads + Gmail thread/message search + attachment download (OAuth, offline refresh token)
- **Slack** (`mcp-servers/slack/`) — Channel search, message search with file metadata, file download (Bot token)
- **Lowercarbon** (`mcp-servers/lowercarbon/`) — Portfolio documents, investor updates, IC memos, investment data (Vectorize, JSON-RPC proxy)
- **Granola** (`mcp-servers/granola/`) — Past meeting notes/transcripts search and retrieval (OAuth token)
- **Affinity** (`mcp-servers/affinity/`) — Company search, Deal Log pipeline data, internal notes (REST API v2, Bearer token)

Each server is conditionally registered — only starts if its credentials exist. All data access is read-only. The agent's only write capability is delivering briefings and file attachments to the user's Slack DM.

## Setup (per user)

### One-time Google auth
```bash
npx tsx scripts/google-auth.ts \
  --credentials groups/<group>/reference/google-credentials.json \
  --token groups/<group>/reference/google-token.json
```
Opens browser for OAuth consent. Token persists indefinitely (Internal Workspace app).

### Required reference files
Place in `groups/<group>/reference/`:
- `google-credentials.json` — GCP OAuth Desktop client (from Cloud Console)
- `google-token.json` — generated by auth script
- `slack-bot-token.txt` — Slack bot token (`xoxb-...`, requires `files:read` and `files:write` scopes for attachment surfacing)
- `affinity-api-key.txt` — Affinity API key (from Settings > Developer > API Keys)
- `granola-token.json` — generated by Granola auth script (optional)

### MCP config
Place in `groups/<group>/.mcp.json`:
```json
{
  "mcpServers": {
    "lowercarbon-mcp": {
      "url": "https://agents.vectorize.io/api/agents/<AGENT_ID>/mcp",
      "headers": {
        "Authorization": "Bearer <VIO_API_KEY>"
      }
    }
  }
}
```

### Agent instructions
Place in `groups/<group>/CLAUDE.md` — contains meeting classification logic, context gathering tiers, output format rules, and behavioral instructions.

## Meeting Classification

The agent classifies calendar events before deciding what to research:

| Type | Detection | Briefing |
|------|-----------|----------|
| SKIP | Events, conferences, demo days, personal entries, holds, reminders | No briefing |
| INTERNAL | All attendees on LC domains, no specific agenda | No briefing |
| PORTFOLIO | Attendee domain or title matches portfolio company (confirmed via `get-portfolio-facts`) | Key updates + open threads |
| LP | Attendee in LP CSV or title heuristics | Context + suggested discussion topics |
| NEW_DEAL | Affinity company search + Deal Log check, email inference (excludes investors/banks/corporates). Stage-aware context: early=heavy external, deep=heavy internal, pass=surface reason | Full context + founder backgrounds + deal pipeline data |
| OTHER | Investors, banks, corporate partners | Lightweight brief |

## MCP Servers

Five independent MCP servers in `container/mcp-servers/`, each built into the container image:

| Server | Tools | Auth |
|--------|-------|------|
| **google** | `list_calendars`, `list_events`, `get_event`, `search_threads`, `get_thread`, `list_messages`, `download_attachment` | Google OAuth (`google-credentials.json` + `google-token.json`) |
| **slack** | `search_slack_channel`, `search_slack_messages`, `download_slack_file` | Bot token (`slack-bot-token.txt`) |
| **lowercarbon** | `portfolio_documents`, `get_company_facts`, `get_portfolio_facts` | Vectorize API key (`.mcp.json`) |
| **granola** | `query_granola_meetings`, `list_granola_meetings`, `get_granola_meeting` | OAuth token (`granola-token.json`, auto-refreshed when a refresh token is available) |
| **affinity** | `search_affinity_companies`, `get_deal_log_entry`, `get_affinity_notes` | API key (`affinity-api-key.txt`) |

## Key Design Decisions

- `get-portfolio-facts` (LC MCP) is the authority for company identification, not StandardMetrics
- **NEW_DEAL meetings must NOT call Lowercarbon MCP** — it only contains portfolio company data
- Affinity Deal Log (list ID 205572) is the authoritative source for deal pipeline status — field IDs are hardcoded
- NEW_DEAL context gathering is stage-aware: early stage = heavy external, deep stage = heavy internal, pass = surface reason prominently
- Granola searches per external attendee individually (all meeting types), not grouped
- MCP servers are independent — 5 focused servers, each conditionally registered based on credential existence
- Slack channels use shortened names vs LC MCP legal names — dynamic lookup via `search_slack_channel`, not a static mapping
- Gmail search uses `{from:email from:"Name"}` union syntax to handle people with multiple email addresses
- Calendar invite emails are filtered out of Gmail results
- Briefings are concise (under 60 seconds to read), sources provided on demand
- Agent asks for confirmation when uncertain about meeting classification
- OneCLI proxy bypassed for remote MCP hosts via `NO_PROXY` env var (includes `api.affinity.co`)

## User Onboarding

### Add a new user

```bash
npx tsx scripts/add-user.ts --name caie --slack-channel-id C0XXXXXX
```

This creates `groups/slack_caie/` with shared config (CLAUDE.md, .mcp.json, google-credentials.json, slack-bot-token.txt) and registers the Slack channel. Personal tokens are left empty.

### Per-user authorization

Each user runs these once (on the NanoClaw host machine):

```bash
# Google Calendar + Gmail (required)
npx tsx scripts/google-auth.ts --token groups/slack_caie/reference/google-token.json

# Granola meeting notes (optional, requires paid plan)
npx tsx scripts/granola-auth.ts --token groups/slack_caie/reference/granola-token.json
```

Then restart: `launchctl kickstart -k gui/$(id -u)/com.nanoclaw`

### How multi-user works

- NanoClaw runs a single Slack Socket Mode connection that receives ALL messages to the bot
- Messages are routed by Slack channel ID to the correct group
- Each group runs in an isolated container with only that user's tokens mounted
- No data intermingling — Kyle's container can't see Caie's tokens or vice versa

### Testing vs production

- **Testing:** Create a shared channel (e.g., `#clawbot-caie-test`) so you can observe other users' briefings while tuning
- **Production:** Each user DMs the Clawbot app directly. Briefings and follow-up conversations are private per-user.

### Infrastructure

- NanoClaw must run on an always-on machine (MacBook with sleep disabled for testing, Mac Mini for production)
- The host machine needs Docker running
- Tokens are portable JSON files — migration to Mac Mini is just copying the directory
- A web onboarding page (Tailscale + Express) is planned for remote user authorization without terminal access
